# 测试功能点提取准出标准V2\.0

# 一、文档目的

规范测试功能点提取过程，确保所有测试功能点满足需求全覆盖、UI可追溯、粒度可测试三大核心要求，作为测试设计阶段的准出依据，为测试用例编写、测试执行及缺陷管理奠定基础。

# 二、适用范围

适用于所有软件项目的测试功能点提取活动，包括但不限于：

- 新功能开发测试

- 需求变更测试

- 回归测试范围界定

# 三、核心术语定义

|术语|定义|
|---|---|
|测试功能点|可被单一测试用例完整验证的最小业务原子单元，具备独立业务价值和可验证性|
|需求覆盖率|已提取测试功能点覆盖的需求条目数 / 总需求条目数 × 100%|
|UI可追溯性|每个测试功能点均能对应到UI文档中的具体元素或交互行为|
|粒度合理性|单个测试功能点仅包含一个业务规则，可通过一组输入验证单一预期结果|

# 四、测试功能点提取原则

## 4\.1 业务语义优先原则

- 严格基于需求文档的业务描述提取，禁止按开发实现逻辑（如接口、代码模块）拆分

- 功能点命名采用"动词\+业务对象\+条件"结构（例："手机号11位合法校验"而非"validatePhone"）

## 4\.2 全文档覆盖原则

- 需求文档：覆盖所有功能需求、业务规则、限制条件、计算公式

- UI文档：覆盖所有输入项、按钮、展示状态、交互逻辑

## 4\.3 异常分支显性化原则

- 每个正向业务规则需同步提取对应的异常路径和边界条件

- 权限控制、状态约束、并发场景需单独提取为功能点

## 4\.4 不可再分原则

- 单个功能点无法拆分为两个独立验证的业务单元

- 拆分后不产生仅包含技术实现细节的功能点（如"调用XX接口"）

# 五、测试功能点标准结构（强制字段）

|字段名称|必填|字段说明|示例|
|---|---|---|---|
|功能点ID|✅|全局唯一标识符，格式：TF\-模块缩写\-序号（TF=Test   Function）|TF\-LOGIN\-003|
|模块名称|✅|需求文档一级目录名称|用户中心|
|子模块名称|✅|页面/业务流程名称|用户登录页|
|功能点名称|✅|动词\+业务对象\+条件，清晰描述验证内容|密码错误5次账户锁定|
|业务目标|✅|该功能点解决的核业务问题|防止暴力破解攻击|
|功能描述|✅|系统应执行的具体行为，不含技术实现|连续5次输入错误密码后锁定账户24小时|
|前置条件|✅|执行测试前必须满足的系统/数据状态|账户已注册且未被锁定|
|输出结果|✅|UI展示/接口返回/数据变更的明确结果|锁定提示文案、剩余解锁时间|

# 七、测试功能点准出标准（质量门禁）

## 7\.1 覆盖率标准（必须100%满足）

|检查项|验收标准|验证方式|
|---|---|---|
|需求覆盖率|100%需求条目均有对应功能点|需求追溯矩阵核对|
|UI元素覆盖率|100%UI输入项/按钮/状态均有对应功能点|UI文档逐元素核对|
|异常分支覆盖率|所有业务规则的异常路径均被提取|需求文档异常描述核对|

## 7\.2 粒度标准（必须100%满足）

|检查项|验收标准|验证方式|
|---|---|---|
|单一业务规则|单个功能点仅验证一个业务规则|专家评审|
|可独立验证|无需依赖其他功能点即可完成测试|测试用例预编写验证|
|无技术实现细节|不包含代码、接口等技术描述|非技术人员可读性检查|

## 7\.3 规范性标准（必须100%满足）

|检查项|验收标准|验证方式|
|---|---|---|
|字段完整性|所有强制字段均已填写|自动化脚本校验|
|命名规范性|符合"动词\+业务对象\+条件"结构|命名规范检查|
|ID唯一性|功能点ID全局唯一|去重校验|

## 7\.4 可追溯性标准（必须100%满足）

|检查项|验收标准|验证方式|
|---|---|---|
|需求追溯|每个功能点关联需求ID|追溯矩阵核对|
|UI追溯|每个功能点关联UI元素编号|UI文档映射核对|
|测试可追溯|可明确对应到测试用例设计|测试设计预规划|
